# 产品专场：用 Claude 写 PRD

写 PRD（Product Requirements Document，产品需求文档）是产品经理的核心工作，也是一项耗时耗力的"重体力活"。结构要清晰、逻辑要缜密，还要兼顾技术理解和用户体验。

好消息是：Claude 可以帮你把这项工作从"从零搭建"变成"迭代优化"。本文面向零 AI 经验的产品经理，提供一套完整的、可落地的 Claude 写 PRD 工作流。

::callout{color="amber" icon="i-lucide-lightbulb"}
**核心原则**：Claude 不是你的"代笔"，而是你的**思维搭档（Thinking Partner）**。最好的 PRD 是人类判断力 + AI 结构化能力的结合。
::

---

## 一、写 PRD 前的准备工作

在打开 Claude 之前，先准备好以下信息。准备越充分，AI 输出质量越高。

### 必备信息清单

| 信息项 | 说明 | 示例 |
|---|---|---|
| 产品/功能名称 | 一句话描述你要做什么 | "用户积分兑换系统" |
| 目标用户 | 这个功能为谁而做 | "平台活跃用户，月均登录 8 次以上" |
| 核心问题 | 解决什么痛点 | "用户获取积分后缺乏消耗渠道，导致积分沉淀" |
| 竞品参考 | 2-3 个竞品或对标产品 | "支付宝积分、京东京豆" |
| 技术约束 | 任何已知的技术限制 | "必须复用现有支付系统，不新增接口" |
| 业务目标 | 可量化的成功标准 | "积分兑换率提升 30%，DAU 提升 5%" |

::callout{color="blue" icon="i-lucide-info"}
**新手误区**：很多人跳过准备阶段，直接对 Claude 说"帮我写个 PRD"。结果生成的文档要么太泛，要么偏离实际需求。多花 5 分钟准备，节省 30 分钟返工。
::

---

## 二、四步工作流：与 Claude 协作写 PRD

### Step 1：注入完整上下文

将准备好的信息一次性提供给 Claude。不要逐条追问——AI 在完整上下文下的表现远好于碎片化对话。

**提示词模板：**

```text
你是一位资深产品经理，曾主导过多个成功产品的需求文档撰写。

我需要你协助撰写一份 PRD，以下是背景信息：

【产品/功能】
{产品名称}：{一句话描述}

【目标用户】
{用户画像}

【核心问题】
{解决的痛点}

【竞品参考】
{2-3 个竞品}

【技术约束】
{已知限制}

【业务目标】
{可量化的成功指标}

请基于以上信息：
1. 先向我提出 3-5 个澄清问题，帮助我补充关键信息
2. 然后输出 PRD 的完整框架
3. 每个部分用结构化格式呈现
```

**Claude 可能的回应：**

- "这个功能是全新模块还是现有功能的增强？"
- "目标上线时间是什么？"
- "是否需要考虑多端适配（Web/App/小程序）？"
- "积分兑换的商品范围是由运营配置还是固定列表？"

::callout{color="green" icon="i-lucide-check-circle"}
**技巧**：Claude 的追问往往比你预想的更全面。不要嫌烦——回答这些问题后，PRD 的完整度会大幅提升。
::

---

### Step 2：苏格拉底式提问（可选但推荐）

如果你对产品方向还不够清晰，可以让 Claude 用结构化问题帮你"梳理思路"。

**提示词模板：**

```text
在正式写 PRD 之前，我想先梳理清楚产品思路。

请针对以下功能，用苏格拉底式提问法帮我逐一审视：

【功能】{功能名称}

请从以下维度提问，每个维度 1-2 个问题：
1. 用户价值：这个功能真的解决了用户的核心痛点吗？
2. 业务价值：与公司战略目标的对齐程度如何？
3. 可行性：技术实现上是否有重大风险？
4. 优先级：为什么是现在做，而不是三个月后再做？
5. 成功标准：如何证明这个功能做成功了？

每个问题请附带"为什么这个问题重要"的简短说明。
```

---

### Step 3：分模块撰写 PRD

不要一次性要求 Claude 输出完整 PRD。按模块逐步推进，每次聚焦一个部分，质量更高。

**推荐撰写顺序：**

1. 文档概述（文档名称、版本、作者、修订记录）
2. 产品目标与背景（背景说明 + 产品目标）
3. 用户与使用场景（用户角色 + 使用场景）
4. 功能清单（功能列表 + 优先级）
5. 功能流程图（主流程 + 分支流程）
6. 页面说明与交互细节（按页面拆分）
7. 业务规则与边界条件
8. 验收标准（必须可量化）
9. 非功能性需求（性能、兼容性、安全性）
10. 异常与错误提示

**单模块提示词模板：**

```text
我们来撰写 PRD 的【功能清单】部分。

背景信息：
{重复之前提供的背景信息}

请输出：
1. 功能清单表格，包含：编号、功能名称、功能描述、优先级（P0/P1/P2）
2. 按 Must / Should / Could 分类整理
3. 每个功能标注预估复杂度（高/中/低）
4. 识别出可能的功能范围蔓延点

输出格式：Markdown 表格 + 文字说明
```

---

### Step 4：AI 辅助审查

初稿完成后，用 Claude 做三轮审查，这是**最容易被忽视但价值最高**的环节。

#### 审查 1：完整性检查

```text
请审查以下 PRD，从三个视角指出可能遗漏的关键信息：

【技术视角】开发团队需要知道但文档中可能缺失的内容
【业务视角】产品经理和决策者需要确认但未明确的假设
【用户视角】最终用户可能遇到但文档未覆盖的场景

PRD 内容：
{粘贴你的 PRD}
```

#### 审查 2：一致性检查

```text
请检查以下 PRD 的一致性：

1. 术语是否前后统一？（例如：同一功能不能前面叫"积分"后面叫"金币"）
2. 逻辑是否自洽？（例如：功能 A 的输入不能和功能 B 的输出矛盾）
3. 目标与功能是否对齐？（每个功能是否都能追溯到业务目标）

PRD 内容：
{粘贴你的 PRD}
```

#### 审查 3：清晰度检查

```text
假设你是新加入团队的工程师，第一次看这份 PRD。

请指出：
1. 哪些地方表述模糊、可能有多种理解？
2. 哪些技术术语需要补充解释？
3. 哪些流程描述缺少关键判断条件？

PRD 内容：
{粘贴你的 PRD}
```

## 三、PRD 提示词速查表

以下是 5 个高频场景的直接可用提示词，复制粘贴即可使用。

### 场景 1：从零生成完整 PRD

```text
你是资深产品经理，请为以下功能撰写完整 PRD：

功能名称：{功能名称}
产品阶段：{探索期 / MVP / 迭代优化}
目标用户：{用户画像}
核心问题：{痛点描述}
竞品参考：{竞品名称}
约束条件：{技术/时间/资源限制}
成功指标：{可量化目标}

PRD 需包含：
1. 文档概述（版本、作者、修订记录）
2. 产品目标与背景
3. 用户角色与使用场景
4. 功能清单（含优先级）
5. 核心流程图（用文字描述）
6. 页面交互说明
7. 业务规则与边界条件
8. 验收标准（必须量化）
9. 非功能性需求
10. 异常处理与错误提示

要求：
- 语言简洁专业，避免冗长
- 验收标准必须包含具体数值
- 边界条件要覆盖常见异常情况
```

### 场景 2：已有零散想法，需要结构化

```text
我有一些零散的产品想法，需要整理成结构化 PRD。

原始素材：
{粘贴你的想法、会议记录、用户反馈等}

请帮我：
1. 提取核心需求和优先级
2. 补充缺失的信息（告诉我需要补充什么）
3. 输出初步的 PRD 框架
4. 识别潜在的风险和依赖
```

### 场景 3：PRD 迭代优化

```text
请审查并优化以下 PRD：

【优化重点】
{清晰度 / 完整性 / 技术可行性 / 用户体验}

【PRD 原文】
{粘贴现有 PRD}

请输出：
1. 问题清单（按严重程度排序）
2. 修改建议（具体到段落或句子）
3. 优化后的完整版本
4. 变更说明（改了什么、为什么改）
```

### 场景 4：生成用户故事

```text
基于以下功能描述，生成标准化的用户故事：

功能：{功能描述}

要求：
- 按 "作为【角色】，我希望【功能】，以便【价值】" 格式
- 每个用户故事附带验收标准（Given-When-Then 格式）
- 区分主流程和异常流程
- 标注优先级
```

### 场景 5：生成竞品分析

```text
请为以下产品做竞品分析，输出可直接放入 PRD 的竞品对比章节：

我方产品：{产品名称}
核心功能：{功能描述}
竞品：{竞品 A}、{竞品 B}、{竞品 C}

分析维度：
1. 核心功能对比
2. 用户体验差异
3. 定价/商业模式
4. 技术优势
5. 我方差异化机会

输出格式：对比表格 + 文字总结
```

## 四、质量检查清单

AI 生成的 PRD 容易出现以下问题，提交给团队前请逐项检查。

### 必查项（阻塞性问题）

- [ ] **目标可量化**：业务目标包含具体数字（如"提升 30%"而非"提升用户体验"）
- [ ] **验收标准明确**：每个功能都有可测试的验收条件
- [ ] **边界条件覆盖**：空状态、异常流程、极限值已说明
- [ ] **术语统一**：同一概念在全文档中用词一致
- [ ] **技术约束确认**：涉及的技术方案已与研发团队初步对齐

### 加分项（提升专业度）

- [ ] **用户故事完整**：主流程 + 异常流程 + 边缘场景
- [ ] **非功能需求**：性能、兼容性、安全性已说明
- [ ] **埋点设计**：关键行为有数据追踪方案
- [ ] **风险识别**：已知风险和应对预案已列出
- [ ] **依赖关系**：外部依赖和阻塞点已标注

### AI 特有问题

- [ ] **不过度理想化**：AI 倾向于忽略现实约束，确认技术可行性
- [ ] **不遗漏负面路径**：AI 容易聚焦"happy path"，检查异常处理
- [ ] **不虚构数据**：AI 可能编造用户研究数据，替换为真实数据
- [ ] **不泛化表述**：将"提升用户满意度"改为可量化指标

## 五、常见陷阱与应对

::card-group
  :::card
  ---
  title: PRD 感觉"太泛"
  icon: i-lucide-file-warning
  ---
  **原因**：提供给 Claude 的上下文不足，或需求描述太抽象。

  **应对**：
  - 补充具体的公司背景、用户研究数据、竞品分析
  - 使用 @-mention（Claude Code 中）引用相关文档
  - 添加用户原话或真实场景案例
  :::

  :::card
  ---
  title: 验收标准不量化
  icon: i-lucide-ruler
  ---
  **原因**：AI 默认使用模糊表述，不会主动要求量化。

  **应对**：
  - 在提示词中明确要求"验收标准必须包含具体数值"
  - 审查时逐一检查每个验收条件是否可测试
  - 参考："页面加载时间 ≤ 2s" ✓ vs "页面加载快" ✗
  :::

  :::card
  ---
  title: 忽略边界条件和异常
  icon: i-lucide-alert-triangle
  ---
  **原因**：AI 倾向于描述"理想流程"，容易遗漏异常分支。

  **应对**：
  - 专门用一个提示词让 Claude 审查异常场景
  - 自问：网络中断怎么办？数据为空怎么办？用户重复操作怎么办？
  - 在功能清单中增加"非目标"列，明确排除范围
  :::

  :::card
  ---
  title: 过度依赖 AI，丧失判断力
  icon: i-lucide-brain-circuit
  ---
  **原因**：AI 输出看起来"专业"，让人产生权威幻觉。

  **应对**：
  - 记住：你对业务的理解永远比 AI 深
  - AI 负责"结构化表达"，你负责"业务判断"
  - 关键决策（优先级、范围、技术方案）必须人工确认
  :::
::

## 六、完整示例：积分兑换系统 PRD

以下是一个通过 Claude 协作生成的 PRD 片段示例，供参考。

### 文档概述

| 字段 | 内容 |
|---|---|
| 文档名称 | 用户积分兑换系统 - 产品需求文档 |
| 版本 | v1.0 |
| 作者 | 产品经理张三 |
| 创建时间 | 2026-04-29 |
| 评审状态 | 待技术评审 |

### 产品目标与背景

**背景说明**：
平台现有积分获取渠道丰富（签到、消费、任务），但积分消耗渠道单一（仅可抵扣少量订单金额），导致用户积分大量沉淀。调研显示，73% 的用户积分余额超过 1000 分但未使用。

**产品目标**：
1. 上线积分兑换商城，提供实物商品、虚拟权益、优惠券三类兑换品
2. 积分兑换率在上线后 3 个月内达到 15%（当前 < 1%）
3. 积分相关页面 DAU 提升 20%

### 功能清单

| 编号 | 功能名称 | 功能描述 | 优先级 | 复杂度 |
|---|---|---|---|---|
| F-01 | 兑换商城首页 | 展示可兑换商品列表，支持分类筛选和搜索 | P0 | 中 |
| F-02 | 商品详情页 | 展示商品信息、积分价格、库存、兑换规则 | P0 | 低 |
| F-03 | 积分兑换流程 | 确认兑换 → 扣除积分 → 发放权益 | P0 | 中 |
| F-04 | 兑换记录查询 | 用户查看历史兑换记录和物流状态 | P1 | 低 |
| F-05 | 积分不足引导 | 积分不足时引导用户获取积分 | P1 | 低 |

### 验收标准（示例）

**F-01 兑换商城首页**
- 页面首屏加载时间 ≤ 1.5s（3G 网络环境下）
- 支持按「实物 / 虚拟 / 优惠券」三级分类筛选
- 空状态：当某分类下无可兑换商品时，显示"暂无商品，去看看其他分类"
- 商品卡片展示：封面图、名称、积分价格、库存状态（充足/仅剩 X 件/已兑完）

## 七、下一步

掌握本文的工作流后，你可以：

1. **建立自己的 PRD 模板库**：将常用提示词保存为 Claude Projects 中的模板
